LAB-03 : Amazon Elastic Load Balancing (ELB)

Elastic Load Balancing (ELB) est un service d'quilibrage de charge d'AWS qui rpartit automatiquement votre trafic entrant sur plusieurs cibles (par exemple, des instances EC2, des conteneurs, etc.) dans plusieurs zones de disponibilit. Il garantit ainsi une meilleure disponibilit de votre application.

Le service ELB prend en charge quatre types dquilibreurs de charge :

  • Application Load Balancers
  • Network Load Balancers
  • Gateway Load Balancers
  • Classic Load Balancers.

Vous pouvez choisir le type d'quilibreur de charge qui correspond le mieux vos besoins. Ce LAB traite les quilibreurs de charge d'application (Application Load Balancers).

Principe de fonctionnement

[Suivant | Haut]

Un quilibreur de charge constitue le point de contact unique pour les clients. Ces derniers envoient leurs requtes l'quilibreur de charge, qui les envoie aux cibles (instances EC2 dans votre cas).

Pour configurer votre quilibreur de charge, vous devez :

  • crer des couteurs (listeners) pour recevoir les requtes de connexion des clients l'aide du protocole et du port que vous avez configurs et dfinir des rgles qui dterminent la manire dont l'quilibreur de charge achemine les demandes vers les cibles dans un ou plusieurs groupes cible. Chaque rgle comprend une priorit, une ou plusieurs actions et une ou plusieurs conditions. Lorsque les conditions d'une rgle sont satisfaites, ses actions sont effectues.
  • crer des groupes cible (targets group) dans lesquels vous allez enregistrer par la suite les cibles. Chaque groupe cible achemine les requtes vers une ou plusieurs cibles enregistres l'aide du protocole et du numro de port que vous spcifiez. Vous pouvez ajouter et supprimer des cibles votre quilibreur de charge au fur et mesure que vos besoins changent, sans interrompre le flux de requtes global vers votre application.

L'quilibreur de charge surveille continuellement l'tat de sant de toutes les cibles enregistres auprs du groupe cible (ce dernier utilise les paramtres de vrification de l'tat par dfaut, sauf si vous les remplacez lors de la cration du groupe cible ou que vous les modifiez ultrieurement). L'quilibreur de charge achemine les requtes uniquement aux cibles saines.

Dans cette section vous allez prparer votre infrastructure rseau en implmentant l'architecture suivante : un VPC personnalis avec quatre sous-rseaux (deux publics et deux privs) rpartis sur deux zones de disponibilit de sorte que chaque zone contient un sous-rseau public et un sous rseau-priv.

Vous devez spcifier les sous-rseaux publics dau moins deux zones de disponibilit afin daccrotre la disponibilit de lquilibreur de charge. Vous pouvez spcifier un seul sous-rseau par zone de disponibilit.

Lquilibreur de charge achemine le trafic jusquaux cibles (instances EC2 dans votre cas) dans les sous rseaux privs de ces zones de disponibilit.

Pour simplifier, lancez une seule instance EC2 dans le sous-rseau priv de chaque zone de disponibilit et crez un seul hte bastion pour vous connecter en SSH ces instances.

Pour les instances se trouvant dans les sous-rseaux privs, veuillez spcifier les donnes utilisateur (User data) permettant d'automatiser l'installation de fastapi, la cration du fichier main.py (vu dans le LAB AWS-EC2) et le lancement du serveur uvicorn. Assurez-vous que le groupe de scurit de ces instances autorise l'accs SSH et l'accs HTTP sur le port 8000.

Cration du Load Balancer

[Prcdent | Suivant | Haut]

Dans cette section vous allez crer et configurer votre quilibreur de charge. Pour ce faire :

Ouvrez la console du service EC2 et slectionnez "Load Balancers" dans le volet de navigation gauche puis cliquez sur "Create Load Balancer". Pour ce LAB, choisissez l'quilibreur de charge de type "Application Load Balancer" (ALB) et cliquez sur le bouton "Create" correspondant.

1. Donnez un nom votre ALB et conservez les valeurs par dfaut pour la mthode et le type dadresse IP :

  • Scheme : internet-facing
  • IP address type : ipv4

2. Sous "Network mapping", slectionnez votre VPC puis cochez les deux zones de disponibilit que vous avez utilises dans votre architecture et pour chaque zone, slectionnez le sous-rseau public.

3. Pour le groupe de scurit dfinissez les rgles permettant d'accepter le trafic HTTP sur le port 80.

4. Pour "Listeners and routing" choisissez le protocol HTTP et le port 80 (valeurs par dfaut) et cliquez sur "Create target group". Un autre onglet s'ouvre pour spcifier les informations de votre groupe cible :

a. Choisissez le type "Instance" (par dfaut) et donnez un nom votre nouveau groupe cible

b. Conservez le protocole par dfaut (HTTP) et spcifier le port (8000). Votre quilibreur de charge acheminera les requtes vers les cibles de ce groupe cible selon le protocole et le port spcifis

c. Slectionnez votre VPC, conserver la version du protocole en tant que HTTP1 et laissez les configurations par dfaut pour les "Health checks" (Vrifications de l'tat) puis cliquez sur "Next".

d. Dans l'interface suivante cochez les instances que vous avez cres dans vos sous-rseaux privs et cliquez sur "Include as pending below" pour les enregistrer comme des cibles auprs de votre groupe cible. Puis cliquez sur "Create target group".

5. Revenez l'onglet de cration du Load Balancer et choisissez votre groupe cible comme "Default action" sous "Listeners and routing" (actualisez la liste si ncessaire) puis cliquez sur "Create load balancer".

Notez bien que vous pouvez crer votre target group l'avance puis le slectionner lors de la cration de votre load balancer.

AWS Auto Scaling

[Prcdent | Suivant | Haut]

Il est recommand d'utiliser le service AWS Auto Scaling dans le cas d'une application qui utilise une ou plusieurs ressources dimensionnables et dont la charge de travail est variable (par exemple une application Web d'e-commerce recevant un trafic variable).

AWS Auto Scaling dimensionne automatiquement les ressources en fonction de la stratgie de dimensionnement que vous avez choisie afin que vous puissiez maintenir les performances et ne payer que pour les ressources dont vous avez rellement besoin.

Si vous devez seulement dimensionner des groupes d'instances EC2, vous allez crer et configurer EC2 Auto-scaling Group via la console EC2 (hors du service AWS Auto Scaling).

Pour chaque EC2 Auto Scaling Group, vous pouvez spcifier la capacit souhaite, le nombre minimum d'instances (le groupe ne descende pas au-dessous de cette taille) et le nombre maximum d'instances (le groupe ne dpasse pas cette taille). Les stratgies de dimensionnement que vous dfinissez ajustent le nombre d'instances, entre le nombre minimum et maximum d'instances, en fonction des critres que vous spcifiez.

Un EC2 Auto Scaling Group dmarre en lanant suffisamment d'instances pour atteindre la capacit souhaite. Il maintient le nombre d'instances en ralisant des vrifications priodiques sur l'tat des instances du groupe. Si une instance devient dfectueuse, le groupe la rsilie et lance une autre pour la remplacer.

[Prcdent | Suivant | Haut]

Les tapes suivantes vous guideront travers la cration et la configuration d'un EC2 Auto-Scaling Group:

Cration d'un modle de lancement (Launch template)

  1. Ouvrez la console Amazon EC2, et choisissez "Launch templates" dans le panneau de navigation gauche, puis cliquez sur "Create launch template"
  2. Donnez un nom et ventuellement une description votre modle de lancement et cochez la case sous "Auto Scaling guidance"
  3. Pour "Amazon machine image (AMI)", choisissez Amazon Linux 2 (HVM); pour "Instance type", choisissez t2.micro et pour "Key pair (login) ", choisissez une paire de cls existante
  4. Sous "Network settings" configurez convenablement votre groupe de scurit
  5. Sous "Advanced details", spcifiez les donnes d'utilisateur (User data) permettant d'automatiser l'installation de fastapi, la cration du fichier main.py (vu dans le LAB AWS-EC2) et le lancement du serveur uvicorn, puis cliquez sur "Create launch template".

Cration d'un EC2 Auto Scaling Group

  1. Toujours dans la console Amazon EC2, choisissez "Auto Scaling groups" dans le panneau de navigation gauche, puis cliquez sur "Create Auto Scaling group"
  2. Donnez un nom votre Auto-Scaling Group et slectionnez le modle de lancement que vous avez cr pour "Launch template". Sinon vous pouvez cliquer sur "Create a launch template" pour la cration d'un nouveau modle de lancement. Choisissez la valeur "Latest" pour "Version" puis cliquez sur "Next"
  3. Dans cette tape de configuration de paramtres indiquez votre VPC et vos deux sous-rseaux privs puis cliquez sur "Next"
  4. Choisissez l'option "Attach to an existing load balancer", puis pour "Existing load balancer target groups" slectionnez votre groupe cible (target group). Sous "Health checks" cochez "ELB" puis cliquez sur "Next"
  5. Pour "Group size" donnez la valeur "2" pour les capacits: minimale, maximale et souhaite. Dans ce LAB vous n'allez pas choisir une stratgies de mise l'chelle, gardez donc le choix "none" coch par dfaut puis cliquez sur "Next"
  6. Passez les tapes "Add notifications" et "Add tags"
  7. Dans la dernire tape "Review", vrifiez votre configuration puis cliquez sur "Create Auto Scaling group"
